## 🎯 The Mission
**Can motion sensors alone reveal whether you scrolled, zoomed, or typed?**

This project explores the genuine research field of **behavioral biometrics**. The goal is to build a single, self-contained HTML/JS mobile web application featuring three distinct mini-screens: a scrolling **Feed**, a pinch-to-zoom **Image**, and a **Typing** box. 

As a volunteer interacts with each screen, the application simultaneously captures their touch events (coordinates, finger count) and the device's motion sensors (accelerometer and gyroscope). By gathering and automatically labeling this combined telemetry, we aim to answer the core analytics question: *Does each gesture leave a distinct, recognizable motion signature in the user's hand?*

---

## 🏗️ Application Architecture: The 3 Capture Screens

The web application simulates three common mobile interfaces to capture natural hand behaviors. Each screen auto-labels its own gestures during data collection.

| Screen | User Interface | Captured Telemetry & Auto-Labels |
| :--- | :--- | :--- |
| **📱 FEED** | A long, scrollable list of items | **Scroll Up / Scroll Down:** Touchmove direction, distance, velocity, and finger count. |
| **🔍 ZOOM** | One large image, pinch-to-zoom | **Zoom In / Zoom Out:** 2-finger distance change over time. |
| **⌨️ TYPE** | A sentence to copy + a text box | **Typing:** Tap positions, inter-key timing, and taps per second. |

> **Note on Sensors:** On ALL screens simultaneously, the **Accelerometer** (x, y, z) and **Gyroscope** (alpha, beta, gamma) stream at the device's native rate, synchronized with exact timestamps.

## 📊 Data Schema
The application exports the captured session data into a standardized CSV format for downstream analytics. Below is the exact schema expected:

```csv
timestamp, screen, event_type, x, y, finger_count, accx, accy, accz, rotAlpha, rotBeta, rotGamma, gesture_label, volunteer_id
```

**Schema Definitions:**
*   `event_type`: `touchstart` | `touchmove` | `touchend` | `motion`
*   `gesture_label`: `scroll_up` | `scroll_down` | `zoom_in` | `zoom_out` | `typing`
*   `volunteer_id`: Anonymized IDs only (e.g., `V01`, `V02`). **Never use real names.**

## 🚀 Development Roadmap: The 6-Prompt Ladder
This project is built iteratively using an AI prompt ladder. Build one stage, **TEST ON YOUR PHONE**, and then proceed to the next.

1.  **P1 - Skeleton:** Create a self-contained mobile HTML file with 3 tabs (Feed, Zoom, Type), a Start/Stop Recording button, and an active tab status bar. Ensure large, readable fonts.
2.  **P2 - Touch:** Enable recording on all tabs. Capture `touchstart`, `touchmove`, and `touchend` with `x, y`, finger count, and timestamp into a primary data array, tagging records with the active tab.
3.  **P3 - Motion:** Integrate `DeviceMotion` capture (accelerometer and rotation rate). Append this to the primary array. Include an **"Enable Sensors"** button that explicitly calls `DeviceMotionEvent.requestPermission()` for iOS compatibility.
4.  **P4 - Auto-Label & Live Stats:** Implement auto-labeling logic (e.g., 2 fingers on Zoom = zoom by distance; vertical touchmove on Feed = scroll; taps on Type = typing). Display live counts for each label.
5.  **P5 - Visualise:** Integrate live analytics. Add a line chart showing accelerometer magnitude for the most recent gesture, a bar chart of gesture counts, and a visual 5-stage pipeline strip (*Capture -> Clean -> Analyse -> Visualise -> Export*) indicating the current active stage.
6.  **P6 - Export:** Add functionality to download the dataset as `data.csv` and `data.json` matching the exact schema. Prompt the user for an anonymized volunteer ID (e.g., `V01`) before each session begins.

## 📈 The Analytics Challenge
The dataset generated by this app will fuel multiple analytical stages throughout the syllabus:

*   **Unit 1 (Descriptive Analytics):** Compute mean, variance, and peak count of acceleration per gesture. Create a comparison table. *Do scroll, zoom, and typing produce visibly different numbers?*
*   **Unit 2 (Classification):** Feature engineering (mean acceleration, variance, duration, dominant axis). Predict the gesture from motion data **ALONE**. Hide the actual label, make a prediction, and calculate model accuracy.
*   **Unit 4 (Clustering):** Collect data from 10+ volunteers. Apply K-Means clustering. *Does each person's hand pattern form its own cluster? Can unsupervised learning tell people apart solely by how they hold their phone?*

## ⚠️ Rules of the Game & Constraints

Understanding these constraints is a core part of the learning process:

1.  **Hosting & HTTPS:** Mobile sensors require a secure context. You must host the app for FREE on GitHub Pages (`github.com` > `new repo` > `upload file` > `Settings` > `Pages`).
2.  **iOS Permissions:** iPhones strictly require user interaction (a tap) to grant motion sensor access. The "Enable Sensors" button is mandatory.
3.  **The Browser Sandbox:** A web page can only see touches *inside* itself—it cannot spy on native apps like WhatsApp. We simulate app screens to gather this data ethically. Understanding this security wall earns marks.
4.  **Consent & Privacy:** Consent is mandatory. Volunteers must agree before data is collected. All data must be stored using anonymized IDs (`V01`, `V02`). Target: 10+ volunteers, with 5 repetitions of each gesture per screen.
5.  **Audit Trail:** Keep your full AI prompt log. Every prompt and bug fix serves as your engineering diary and must be submitted by the Group Lead.
